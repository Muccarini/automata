## Design Context

### Users
Solo developers and makers building their own integrations and automations. They work in focused sessions, value precision and productivity, and are comfortable with technical interfaces. The tool is a power-user instrument, not an onboarding experience.

### Brand Personality
Dark · Technical · Utilitarian

The interface should feel like a precision instrument — purposeful, confident, and uncluttered. Not warm, not loud, not friendly. Think VS Code's editor, Vercel's deployment dashboard, or Linear's project board. Every element earns its place.

### Aesthetic Direction
- Always dark mode
- Near-monochromatic with extremely restrained color accents (dim, low-opacity tints only)
- No colored borders or backgrounds on interactive containers — color is reserved for tiny semantic indicators
- Sharp, functional typography with minimal letter spacing on micro-labels
- Anti-references: avoid n8n-style colorful node cards, Zapier-style rounded consumer aesthetic

### Design Principles
1. **Restrain color to indicators only.** Color communicates type or state — it never decorates. A node border has no color; a node-type dot does.
2. **Reduce until it hurts, then stop.** Every visual element must justify its existence. Borders, shadows, and backgrounds at minimum effective contrast.
3. **Let hierarchy breathe.** Spacing and weight create structure. Don't use color to compensate for weak hierarchy.
4. **Utilitarian readability.** Dense information displays are expected. Optimize for scanning, not impression.
5. **Functional motion only.** No decorative animations. State transitions are instant or barely-there.
